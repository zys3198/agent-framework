# agent-framework 优化评估

> 基于 [面试记录-结构化.md](面试记录-结构化.md) 的 5 条改进意见，逐条对照 `agent_framework/` 实际代码核实。
> 其中有一条是面试官没挖到、但代码里真实存在的 P0 bug，放在最前面。

---

## 先说最严重的：memory 在工具循环里根本没注入（面试官没挖到的 P0）

这是在对照第 4、5 条意见时翻代码挖出的真问题，比面试官指出的还严重。

`build_system_prompt()` 把 todos/plan/lessons 注入 system prompt，但**它只在 DIRECT 路径被调用**（`runtime/agent.py`）。而所有真正调工具的路径（`SIMPLE_TOOL`、`PLAN_REQUIRED` 的每一步）走的是 Executor，Executor 构造请求时是这么写的（`runtime/executor.py`）：

```python
session.messages.append(Message(role="user", content=prompt))
messages: list[dict[str, Any]] = [m.to_dict() for m in session.messages]
tools = self._registry.schemas()
```

直接拿 `session.messages` 拼，**全程没有 system prompt，也没有 memory**。而 `session.messages` 里永远不含 system 消息（`session/models.py` 的 `Message.role` 只有 `user | assistant | tool`），所以 Executor 发给 LLM 的请求里：

- todos 看不到
- plan 看不到
- lessons 看不到

ReWOO 三个子步骤（`_plan_dag` / `_execute_dag` / `_solve`）同样不注入 memory。

这直接把第 4 条意见里本来"做对了"的部分废掉了——todo 虽然外化进了 memory 并持久化，但**模型在工具循环里压根读不到自己的 todo 列表**，除非它每轮主动再调一次 `todo list`。面试官说"todo 只返回 string 给模型"，实际情况是连"从 system prompt 稳定拿到 todo"都没做到。

**这必须第一个修**：Executor 和 ReWOO 构造 messages 时统一 prepend `build_system_prompt(memory)`。修了它，第 4 条的"双通道加强注意力"才有一个能落地的承载点。

## 逐条对照面试官 5 条意见

### 意见 1 — function calling schema 加载位置上移（工具层/客户端）

**现状**：schema 定义在工具类（`Tool.parameters` ClassVar，合理），但加载和组装散在两层——Executor 里 `self._registry.schemas()`（`runtime/executor.py`）取出来，再传给 `LLMClient.chat_with_tools`，client 又包一层 `{"type":"function","function":t}`（`llm/client.py`）。

**问题**：Executor 同时是"协议加载点"和"执行循环"，职责混在一起；协议描述（`type: function` 这层壳）漏在 LLMClient 里。

**优化方向**：把"工具协议描述"上移成一个独立的 `ToolSpec`/schema 视图层（可由 ToolRegistry 统一渲染成 OpenAI tools 数组），Executor 只负责 dispatch。这条相对最轻，代码本身能跑，属于架构整洁度问题。

### 意见 2 — 路由小模型思路挪去记忆召回（意图识别 + 关键词 + 向量库）

**现状**：Router 是面试官批评的核心靶子，而且批评成立。看 `runtime/router.py:33`：

```python
async def classify(self, user_input: str, memory: Memory) -> Route:
    messages = [{"role": "system", "content": _ROUTER_PROMPT}]
```

`memory` 参数**收了但完全没用**，system prompt 是死的 `_ROUTER_PROMPT`，没有任何上下文、历史、todos。路由模型 A 拿到的就是一句"把输入分成三类"，它怎么判断"自己能否完成"？它连上文在干嘛都不知道。面试官那句"A 评不了自己能否完成"在这里是字面成立的——A 根本没被喂过上下文。

**问题**：`memory` 是个死参数（API 设计缺陷，也是 bug 信号）；一个无状态 LLM 猜"难度"，既不稳又烧 token（每轮一次 LLM 调用只换来一个三分类）。

**优化方向**：按面试官说的，路由别让大模型自评难度，改成**廉价召回**：规则/关键词 + 小模型意图识别 + 向量库召回历史，决定走 DIRECT / SIMPLE_TOOL / PLAN_REQUIRED。把 Router 这次 LLM 调用换成确定性召回，是省 token 和提准的双重收益。

### 意见 3 — 持续 ReAct 滚动，替代 plan 后自纠

**现状**：项目把 plan-execute-reflect-replan 做得很重。Reflexion 攒到 3 条 lesson 判"穷尽"（`runtime/reflexion.py` `_EXHAUSTION_THRESHOLD = 3`）→ 升级 Replanner 重规划（`runtime/replanner.py`）→ 还有 ReWOO 的 DAG/solver 链。这正是面试官批评的"plan → execute → reflect → replan"。

**问题**：面试官的原话逻辑——模型刚做完 plan 注意力在顶峰，让它立刻自纠，纠不过来；没完成时它也不知道问题在哪；等完成才发现错等于重做烧 token。当前 Reflexion 的"学 lesson"其实很弱（只往 memory 追加一行字符串，而且因为上面那个 P0 bug，lesson 还没注入工具循环），重规划的收益存疑。

**优化方向**：把"先 plan 再自纠"收敛成**单条持续的 ReAct 循环**（边观察边调），把 Planner/Replanner 降级成"可选的前置分解"而非主路径。ReWOO 这套 DAG 解耦可以作为并行加速保留，但 Reflexion → Replanner 的宏观自纠链值得砍掉或大幅简化。这是减负，不是加功能。

### 意见 4 — todo/工具运行时外化到存储，代码层能召回（lost in the middle）

**现状**：todo 已经外化到 `session.memory.todos` 并持久化（`tools/todo.py` + `session/store.py`），这一步其实比面试官预设的要好。

**问题**有两层：

1. 上面 P0 bug：外化了但工具循环里没注入，等于白外化。
2. 真正的 lost in the middle：**messages 没有任何窗口治理**。Executor 是 `[m.to_dict() for m in session.messages]` 全量拼，ReWOO 同样全量。多轮 ReAct 涨到几十上百条消息后，attention 被稀释，todo 夹在中间找不回。面试官要的"代码层能召回"现在缺一个"模型注意力偏散时主动把当前任务拉回来"的机制（系统重注入或主动召回）。

**优化方向**：P0 先修；然后加最朴素的消息窗口裁剪（保留 system + 最近 N 轮 + 最近 todo 状态），让 todo 状态始终在 system prompt 高权重位置。这才是 lost in the middle 的解法。

### 意见 5 — 改进 memory 召回（借鉴 Claude Code，不做 RAG）

**现状**：`build_system_prompt` 是**全量注入**（todos/plan/lessons 一股脑塞 system prompt），没有任何按相关性筛选、压缩、召回。messages 也是全量拼接。这就是面试官说的"框架没有 memory 的窗口裁剪、压缩、召回策略"。

**优化方向**（这是面试官布置的作业，也是 ROI 最高的一块）：借鉴 Claude Code 的两段式——元信息（摘要/索引）注入 prompt，模型按需用工具拉具体文件；或并行小模型预检索记忆库，在下一轮 ReAct 前注入。明确不做完整 RAG（"那一套搞出来没必要"）。关键动作：**把"全量注入"换成"召回 + 压缩"**。

## 优化优先级

| 优先级 | 项 | 依据 |
|---|---|---|
| P0 | Executor/ReWOO 注入 system prompt（修复 memory 盲区） | 真实 bug，让现有 memory/todo 机制真正生效 |
| P1 | memory 召回 + 消息窗口裁剪（意见 4+5） | 面试作业 + lost in the middle，ROI 最高 |
| P2 | Router 换成廉价召回，废掉死参数 memory（意见 2） | 省 token + 提准，改动可控 |
| P2 | 砍/简化 Reflexion → Replanner，收敛到持续 ReAct（意见 3） | 减负，砍代码 |
| P3 | 工具协议描述上移成独立视图层（意见 1） | 架构整洁，影响最小 |

---

## 一句话总结

面试官的 5 条里，第 4、5 条（记忆/窗口治理）是这个项目真正缺的能力，第 2、3 条（路由、自纠）是过度设计要往回收，第 1 条是整洁度问题；而比这 5 条都更要紧的，是 memory 注入那个 P0——它让 todo/lessons 的外化在工具路径上完全失效。
